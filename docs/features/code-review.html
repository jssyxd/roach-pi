● Overview 개요

ROACH PI provides two code review modes with different depth and cost trade-offs:

ROACH PI는 깊이와 비용 트레이드오프가 다른 두 가지 코드 리뷰 모드를 제공합니다:

  • /review — single-pass review performed directly by the current agent. Fast, lightweight, no subagent overhead. 현재 에이전트가 직접 수행하는 단일 패스 리뷰. 빠르고 가벼우며 서브에이전트 오버헤드 없음.
  • /ultrareview — multi-agent pipeline that dispatches 10 parallel reviewers, a verification pass, and a synthesis step. Thorough but slower. 10개의 병렬 리뷰어, 검증 패스, 종합 단계를 디스패치하는 다중 에이전트 파이프라인. 철저하지만 느림.

▶ /review — Single-Pass Review — 단일 패스 리뷰

Syntax 구문

/review [target]

target can be a PR number, PR URL, branch name, or omitted for auto-detection. No confirmation is required — the review starts immediately.

target은 PR 번호, PR URL, 브랜치 이름, 또는 자동 감지를 위해 생략할 수 있습니다. 확인 없이 리뷰가 즉시 시작됩니다.

Target Validation 타겟 검증

The target is validated against /^[a-zA-Z0-9._/:\-]+$/ before any prompt is sent. Shell metacharacters (;, |, &, >, `, $()) are rejected immediately to prevent injection into the downstream gh / git commands.

타겟은 프롬프트 전송 전에 /^[a-zA-Z0-9._/:\-]+$/로 검증됩니다. 셸 메타문자(;, |, &, >, `, $())는 다운스트림 gh / git 명령어로의 인젝션을 방지하기 위해 즉시 거부됩니다.

Review Output 리뷰 출력

The review is output directly to chat, grouped by file. Each finding includes: what, where (file:line), severity (Critical/High/Medium/Low), and a one-line suggested fix. No file is saved and no subagents are dispatched.

리뷰는 파일별로 그룹화되어 채팅에 직접 출력됩니다. 각 발견 사항은 내용, 위치(file:line), 심각도(Critical/High/Medium/Low), 한 줄 수정 제안을 포함합니다. 파일은 저장되지 않고 서브에이전트도 디스패치되지 않습니다.

■ Diff Commands Diff 명령어

Both /review and /ultrareview use the same target resolution logic:

/review와 /ultrareview 모두 동일한 타겟 해결 로직을 사용합니다:

Target타겟 Command명령어
PR numberPR 번호 gh pr diff <number>
PR URLPR URL gh pr diff <url>
Branch name브랜치 이름 git diff main...<topic>
Auto (PR found)자동 (PR 발견) gh pr list --head <branch> → gh pr diff <number>
Auto (no PR)자동 (PR 없음) git diff main...HEAD + git diff + git diff --cached
💡 Note💡 참고

gh accepts PR numbers and full GitHub PR URLs interchangeably — no special URL parsing is needed.

gh는 PR 번호와 전체 GitHub PR URL을 상호 교환적으로 허용합니다 — 별도의 URL 파싱이 필요 없습니다.

★ /ultrareview — Multi-Agent Pipeline — 다중 에이전트 파이프라인

The ultrareview command dispatches a 3-stage pipeline with up to 12 subagent invocations. A confirmation dialog is required before the pipeline starts.

ultrareview 명령어는 최대 12개의 서브에이전트 호출로 3단계 파이프라인을 디스패치합니다. 파이프라인 시작 전 확인 대화상자가 필요합니다.

Stage 1: Finding (10 parallel reviewers)1단계: 발견 (10개 병렬 리뷰어)

The diff is resolved once and written to a shared artifact (e.g. docs/engineering-discipline/reviews/.tmp/<date>-<topic>.diff). Then 10 subagents are dispatched in parallel — 5 reviewer roles × 2 seeds each. Seed 2 is instructed to focus on findings seed 1 might miss.

diff는 한 번 해결되어 공유 아티팩트(예: docs/engineering-discipline/reviews/.tmp/<date>-<topic>.diff)에 작성됩니다. 그 후 10개의 서브에이전트가 병렬로 디스패치됩니다 — 5개 리뷰어 역할 × 시드 2개. 시드 2는 시드 1이 놓칠 수 있는 발견에 집중하도록 지시받습니다.

Stage 2: Verification2단계: 검증

reviewer-verifier is dispatched in single mode with the aggregated findings from all 10 reviewers. It deduplicates findings, filters false positives, and assigns final severity and confidence ratings.

10개 리뷰어의 모든 집계된 발견과 함께 reviewer-verifier가 단일 모드로 디스패치됩니다. 발견을 중복 제거하고, 위양성을 필터링하며, 최종 심각도 및 신뢰도 등급을 부여합니다.

Stage 3: Synthesis3단계: 종합

review-synthesis is dispatched in single mode with the verified findings. It produces a structured report and saves it to docs/engineering-discipline/reviews/<YYYY-MM-DD>-<topic>-review.md. A 5-item top-priority summary is also streamed to chat.

검증된 발견과 함께 review-synthesis가 단일 모드로 디스패치됩니다. 구조화된 리포트를 생성하고 docs/engineering-discipline/reviews/<YYYY-MM-DD>-<topic>-review.md에 저장합니다. 최우선 5개 항목 요약도 채팅에 스트리밍됩니다.

⚠ Guard⚠ 가드

No agent whose name contains "worker" is ever dispatched in this pipeline — only reviewer-* and review-synthesis are allowed.

이 파이프라인에서는 이름에 "worker"가 포함된 에이전트는 디스패치되지 않습니다 — reviewer-*와 review-synthesis만 허용됩니다.

■ Reviewer Roles리뷰어 역할

Each of the 5 reviewer roles is dispatched twice (seed 1 and seed 2) for complementary coverage:

5개 리뷰어 역할 각각이 상호 보완적 커버리지를 위해 두 번(시드 1과 시드 2) 디스패치됩니다:

Role역할 Focus집중 영역
reviewer-bug Logic errors, boundary conditions, null/undefined, race conditions, missing error handling 논리 오류, 경계 조건, null/undefined, 레이스 컨디션, 누락된 에러 핸들링
reviewer-security Injection, auth, authorization, crypto misuse, data exposure 인젝션, 인증, 인가, 암호화 오용, 데이터 노출
reviewer-performance Algorithmic complexity, unnecessary allocations, sync I/O on hot paths, cache misses 알고리즘 복잡도, 불필요한 할당, 핫 경로의 동기 I/O, 캐시 미스
reviewer-test-coverage Missing tests, happy-path only, uncovered edge cases 누락된 테스트, 해피 패스만 존재, 커버되지 않은 엣지 케이스
reviewer-consistency Convention drift, duplication, missing reuse of existing utilities 컨벤션 드리프트, 중복, 기존 유틸리티의 재사용 누락

● Verification & Synthesis검증 및 종합

reviewer-verifier

Receives the aggregated output from all 10 reviewers. Performs three actions:

10개 리뷰어의 집계된 출력을 받습니다. 세 가지 작업을 수행합니다:

  • Deduplicate — removes findings reported by multiple reviewers for the same location. 중복 제거 — 동일한 위치에 대해 여러 리뷰어가 보고한 발견을 제거합니다.
  • Filter false positives — discards findings that don't hold against the actual codebase. 위양성 필터 — 실제 코드베이스에서 성립하지 않는 발견을 버립니다.
  • Attach ratings — assigns severity (low / medium / high / critical) and confidence to each surviving finding. 등급 부여 — 각 생존 발견에 심각도(low / medium / high / critical)와 신뢰도를 부여합니다.

review-synthesis

Takes the verified findings and produces the final structured review report. Writes to docs/engineering-discipline/reviews/<YYYY-MM-DD>-<topic>-review.md. The report includes a summary, per-file findings ranked by severity, and a top-priority list streamed to chat.

검증된 발견을 받아 최종 구조화된 리뷰 리포트를 생성합니다. docs/engineering-discipline/reviews/<YYYY-MM-DD>-<topic>-review.md에 작성합니다. 리포트에는 요약, 심각도별 순위의 파일별 발견, 채팅에 스트리밍되는 최우선 목록이 포함됩니다.

▶ Choosing Between Modes모드 선택 가이드

▶
/review

Quick feedback on small changes. Single-pass, no subagents, instant output to chat. Ideal for PRs under ~200 lines or when you just need a second pair of eyes.

작은 변경 사항에 대한 빠른 피드백. 단일 패스, 서브에이전트 없음, 채팅에 즉시 출력. 약 200줄 미만의 PR이나 간단한 검토가 필요할 때 적합합니다.

★
/ultrareview

Major PRs, architectural changes, or when thorough coverage is critical. 10 parallel reviewers + verification + synthesis. Takes several minutes but produces a saved, structured report.

주요 PR, 아키텍처 변경, 철저한 커버리지가 중요할 때. 10개 병렬 리뷰어 + 검증 + 종합. 몇 분이 걸리지만 저장되는 구조화된 리포트를 생성합니다.
